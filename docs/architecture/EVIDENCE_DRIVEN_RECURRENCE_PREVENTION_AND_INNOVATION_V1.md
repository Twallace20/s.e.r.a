# Evidence-Driven Recurrence Prevention and Innovation v1

Status: architecture locked, integrated preflight certified, full recurrence-prevention proof pending.

This document locks a cross-cutting Base MVP invariant: S.E.R.A. must be designed so that a materially evidenced and certified failure is not unknowingly repeated under materially equivalent conditions.

This is not the active lesson Runtime implementation. Milestone 13 adds bounded integrated-loop preflight records, but it does not certify lessons, activate prevention rules, automatically promote innovation, or complete the Milestone 14 recurrence-prevention proof.

## Invariant

Every material failure, correction, operator override, regression, and improvement opportunity must be capable of producing an evidence-linked learning signal.

Before materially similar work, S.E.R.A. must eventually retrieve and consider applicable certified lessons, known failure patterns, active prevention rules, certified alternatives, relevant operator overrides, superseded lessons, and proven improvement strategies.

A known certified failure path must not be repeated blindly. S.E.R.A. must block it, warn and require governed review, or select a certified alternative.

## Required Evidence

Reusable lessons require durable evidence, exact source references, applicability boundaries, non-applicability boundaries, reproducibility where required, evaluation, certification, explicit activation or promotion authority, and rollback or supersession support.

One observation is not automatically a certified lesson. Model output is candidate intelligence only. Operator statements can create candidates or overrides, but reusable behavior still requires evidence and authority.

## Learning Lanes

- Acquisition: bring bounded new evidence into S.E.R.A. without claiming it is true or reusable.
- Repair: correct a demonstrated failure while preserving evidence of the original failure.
- Improvement: compare a candidate against a baseline and require measurable evidence before promotion.
- Adaptation: apply a certified pattern to a related context inside declared scope.
- Recurrence prevention: retrieve known failure paths and prevent blind repetition.
- Innovation: propose a materially different approach, capability, authority, dependency, or risk class under stronger governance.

Innovation remains distinct from ordinary learning. It may propose a materially different approach, but it cannot activate itself.

## Context Matching

Exact or materially equivalent known failures require prevention behavior.

Related but different contexts require scoped application with applicability and non-applicability explanations.

Out-of-scope contexts must not be incorrectly blocked. A prevention rule must be narrow enough to avoid suppressing safe unrelated work.

## Operator Overrides

Operator overrides require evidence, authority, reason, scope, and audit records. An override may permit a reviewed exception, but it does not erase the underlying lesson, failure, or prevention rule.

Control Plane retains promotion and override authority.

## Supersession

Lessons may be superseded but not silently erased. Failed lessons, rejected lessons, inactive lessons, active lessons, superseded lessons, and operator overrides must remain inspectable.

## Innovation Boundary

Innovation requires authorization, isolated experimentation, baseline comparison, reproducibility, certification, explicit promotion, and rollback readiness before activation.

No model, capability, Studio, or Desktop surface may promote innovation automatically.

## Base MVP Acceptance

Milestone 11 reserves Desktop Operator surfaces for known failure patterns, applicable lessons, prevention warnings, certified alternatives, operator overrides, superseded lessons, improvement proposals, innovation proposals, supporting evidence, and applicability explanations. These may render honest empty states until the Runtime exists.

Milestone 12 must produce evidence-linked operator corrections, failed-output signals, evaluation failures, improvement opportunities, and innovation opportunities. Interaction history must not automatically become reusable rules.

Milestone 13 runs a learning preflight before capability or approach selection and records bounded preflight evidence for applicable, uncertain, out-of-scope, certified-alternative, and override cases.

Milestone 14 is canonically named: Milestone 14 - Learning Generalization, Recurrence Prevention, and Innovation Proof.

Milestone 15 must prove restart persistence for failure records, certified lessons, prevention rules, lesson scopes, superseded lessons, overrides, and innovation proposals.

Milestone 16 must prove the portable Base MVP can preserve evidence, correct a controlled mistake, certify the correction, restart, retrieve the certified lesson for a materially equivalent request, prevent the known mistake, and explain the evidence in Desktop Operator.

## Milestone 14 Durable Evidence Update

Milestone 14 implements the Learning Governance Runtime proof using migration v11. The schema includes independent records for evidence links, lesson certifications, lesson activations, lesson supersessions, and innovation evidence links. It also records context IDs separately from context hashes, repair versions, versioned lesson scopes, aggregate type, owning service, and Runtime instance ID on learning events.

Supersession does not erase history. Version `1` remains inspectable as `SUPERSEDED`, version `2` is `ACTIVE`, prevention rules point to version `2`, and Integrated Loop durable preflight returns version `2`.
